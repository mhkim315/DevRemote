# 05. 라이선스 및 비즈니스 모델 (License & Business Model)

## 1. 라이선스 전략 (투트랙 아키텍처를 통한 AGPL 한계 극복)
처음 구상했던 'WebRTC 완전 자체 개발(Clean Room)' 전략은 폐기되었습니다. 개발 효율을 극대화하기 위해 **AGPL-3.0을 역이용하는 전략**을 채택합니다.

* **문제 제기:** AGPL 코드의 일부를 가져와 합치면(Linking) 전체가 AGPL로 전염되어 상용화가 불가능해짐.
* **해결책 (프로세스 격리):** 
  * 화면 캡처와 마우스 제어를 담당하는 **RustDesk 코어(Track 1)**는 AGPL 규정에 따라 오픈소스로 투명하게 공개함. (이는 유저에게 보안 신뢰도를 주는 해자로 작용)
  * AI 상태를 관제하고 수익화의 핵심이 되는 **컴패니언 데몬(Track 2)**은 RustDesk와 코드를 섞지 않고 **완전히 독립된 프로세스**로 실행시킴.
  * 독립 프로세스끼리 소켓(Socket)이나 REST API로 통신하는 것은 AGPL 파생 저작물로 간주되지 않음. 따라서 컴패니언 데몬 코드는 **100% 비공개 상용화(Closed-source)**가 가능해짐.

## 2. 오픈소스 해자 (Moat) 전략 (GitLab 모델)
우리는 B2D(Business to Developer) 시장을 타겟팅하므로, 폐쇄적인 수익 모델보다는 **오픈 코어(Open Core)** 모델이 압도적으로 유리합니다.
* **기본판(Free):** P2P 다이렉트 통신, 1대의 데스크톱 제어, 기본적인 에이전트 모니터링 기능은 100% 무료 오픈소스로 Github에 풉니다.
* **이유:** 경쟁사가 유사한 앱을 만들려 해도, 이미 "무료이자 코드가 투명하게 공개되어 백도어 걱정이 없는" 우리 앱을 이길 수 없습니다. 오픈소스 자체가 거대한 진입 장벽(해자)이 됩니다.

## 3. 수익 모델 (Premium Features)
무료 사용자를 생태계에 묶어둔 뒤, 다음 기능들로 B2B 및 헤비 유저에게 과금합니다.
1. **다중 머신 관제 (Multi-Fleet):** 서버 3대 이상에서 돌아가는 에이전트들을 동시 관제.
2. **Team Collaboration:** 에이전트의 승인(Approve) 권한을 팀원들과 공유.
3. **무제한 릴레이 대역폭:** P2P 연결이 안 되는 보안망 환경의 유저를 위한 대역폭 보장.

## 4. 리스크 관리: Firebase API 키 방어
* 모바일 앱 내부에 Firebase 설정 정보(API 키 등)가 하드코딩될 경우, 해커가 이를 뜯어내 악용할 수 있습니다.
* **방어:** Firebase Security Rules를 극한으로 조이고, App Check(디바이스 변조 방지) 기술을 켜서, 실제 구글 플레이/앱스토어에서 정상적으로 다운로드한 우리 앱이 아니면 Firebase DB 접근 자체를 거부시킵니다.
