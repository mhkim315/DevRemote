# 08. 차기 AI 에이전트를 위한 완벽 인수인계서 (Handoff Guide)

환영합니다! 당신은 회원님과 함께 **DevRemote (가칭)** 프로젝트를 이어받아 개발을 진행할 차기 AI 에이전트입니다. 이 문서는 우리가 과거 세션에서 치열하게 논의하고 확정한 프로젝트의 방향성, 타겟 고객, 기술 아키텍처, 그리고 당신이 당장 내일 아침에 실행해야 할 **Next Action Items**를 완벽하게 정리한 가이드입니다.

---

## 1. 프로젝트 정체성 및 타겟 고객 (Market & Target)

* **제품 정의:** AI가 코딩하는 과정을 스마트폰으로 구경하고 통제하는 **B2C 특화 모바일 커맨드 센터 (모바일 리모컨)**
* **핵심 타겟 고객:** 수백만 명의 **B2C 바이브 코더 (Vibe Coders)**
  * **[중요]** 우리의 타겟은 터미널(CLI)에 익숙한 하드코어 개발자가 **아닙니다.** 
  * 2026년 기준, 바이브 코더들은 Cursor, Replit, Bolt.new 같은 화려한 **GUI 환경**에서 AI가 코드를 짜고 브라우저가 새로고침되는 시각적 과정(Vibe)을 즐깁니다.
* **해결하는 페인포인트:** 에이전트에게 작업을 시켜두고 침대에 누웠을 때, "AI가 에러 없이 잘하고 있나?" 궁금해하거나 "이 파일 덮어쓸까요?"라는 AI의 질문에 스마트폰으로 직관적으로 **[승인(Y)]** 버튼을 눌러주기 위함입니다.

---

## 2. 핵심 아키텍처 피봇의 역사 (History of Pivot)

당신이 코드를 짜기 전 반드시 알아야 할 기술적 결정 사항입니다.

* **초기 접근 (Phase 1):** 오픈소스 원격 제어 앱인 `RustDesk`를 포크하여 모바일 UI의 거추장스러운 버튼들을 모두 날리고 코딩용 플로팅 버튼(FAB)을 달았습니다.
* **위기 (AGPL 한계):** RustDesk의 AGPL-3.0 라이선스 때문에 상용화가 막힌다고 판단하여, "WebRTC부터 화면 캡처까지 바닥부터 새로 짜자(Clean Room)"는 제안이 있었습니다.
* **해결 및 최종 결론 (Two-Track 아키텍처):** 밑바닥부터 화면 캡처와 하드웨어 인코딩을 짜는 것은 12~20주가 걸리는 자살 행위임을 깨달았습니다. 더구나 타겟 유저(바이브 코더)는 Cursor의 화려한 GUI 화면을 폰으로 봐야 하므로 압도적인 화면 캡처 엔진이 필수적입니다.
  * **▶ Track 1 (화면 캡처 및 제어망):** 이미 90% 완성된 RustDesk 포크 버전을 그대로 유지합니다. AGPL로 소스를 투명하게 공개하여 오히려 유저 신뢰(보안 해자)를 구축합니다.
  * **▶ Track 2 (AI 컴패니언 데몬):** AI의 상태를 읽어 모바일로 [승인] 팝업을 띄워주는 핵심 비즈니스 로직은 **MIT 라이선스(Go/Rust)**를 활용해 **완전히 독립된 데몬 프로세스**로 신규 개발합니다. 독립 프로세스 통신은 AGPL 전염을 피할 수 있어 100% 상용화가 가능합니다.

---

## 3. 기술 스택 요약

* **Mobile App (Viewer):** Flutter (RustDesk 포크 기반, UI 단순화 완료)
* **Desktop Core (Video/Input):** Rust (RustDesk 포크 기반)
* **Companion Daemon (AI Brain):** Go (pion/webrtc) 또는 Rust (webrtc-rs) 기반으로 **[당신이 새로 개발해야 함]**
* **Signaling/Auth:** Firebase (비용 최소화, E2EE IP 교환)
* **Network:** P2P 직접 연결 (UPnP, STUN 홀펀칭 지원) / 최소한의 Relay 서버 폴백

## 4. 당신의 Next Action Items (내일 출근 시 즉시 실행할 것)

당신이 이 프로젝트에 배정되었다면, 다른 기획을 엎을 필요 없이 다음 순서대로 실무를 진행하십시오.

1. **Phase 1.2 UX 타격감 검증 (모바일):** 
   * 이전 에이전트가 Github Actions를 통해 `devremote-mvp` 브랜치에서 **Build #11 APK**를 성공적으로 뽑아두었습니다. 
   * 회원님께 해당 APK를 안드로이드 폰에 설치해 보실 것을 안내하고, 기존에 만든 `[DevKeypad]` (단일 FAB, 안전 Paste 800ms)의 사용성이 바이브 코딩을 모니터링하기에 적합한지 피드백을 받으십시오.
2. **Phase 2 데몬 프로토타입 뼈대 세우기 (데스크톱):** 
   * UI 피드백이 끝나면, 바로 Track 2 (컴패니언 데몬) 개발에 착수하십시오.
   * Go 언어(추천) 또는 Rust를 사용하여 PC의 백그라운드에서 실행되는 초경량 프로세스 환경을 세팅하십시오.
3. **네트워크 연결 보강 고려:** 현재 RustDesk 구조에서 Tailscale 같은 VPN 없이 Firebase 시그널링만으로 1:1 홀펀칭을 안정적으로 뚫을 수 있도록 네트워크 파이프라인 설계를 시작하십시오.

> **에이전트 행동 수칙:** "새로 짜는 것은 죄악이다." 우리는 타임-투-마켓이 생명인 스타트업입니다. 12줄의 문서 수정보다 1번의 APK 테스트와 1줄의 데몬 코드가 더 가치 있음을 명심하십시오. Good luck!
