# 02. 네트워크 아키텍처 (Network Architecture)

## 1. 아키텍처 원칙: Tailscale 제거 & 서버리스 1:1 다이렉트
복잡한 VPN(Tailscale 등) 설치 없이, 일반 유저도 앱을 깔기만 하면 즉시 자신의 PC와 직통 연결(P2P)되는 플러그 앤 플레이(Plug-and-Play)를 구현합니다.

## 2. Firebase 기반 시그널링 (Signaling) 구조
* **목적:** 변동하는 PC의 공인 IP를 모바일이 알아내기 위한 $0 비용의 연락처 보관함.
* **작동 방식:**
  1. 데스크톱 데몬이 켜지면 자신의 공인 IP와 포트를 Firebase Realtime Database에 업데이트(Heartbeat).
  2. 모바일 앱이 실행되면 Firebase에서 IP를 읽어옵니다.
  3. 이 과정에서 IP 데이터는 E2EE 암호화되어 전송되므로 Firebase 관리자도 사용자의 IP를 알 수 없습니다.

## 3. WebRTC와 P2P 연결 3단계 전략
### Step 1. 로컬 네트워크 (LAN) 감지
모바일과 PC가 동일한 와이파이에 있다면, 외부 인터넷을 거치지 않고 내부망 IP로 즉시 WebRTC 세션을 수립합니다.

### Step 2. UPnP / NAT-PMP 포트 개방 및 현실 점검
* **이상:** 데몬이 공유기(라우터)에 요청하여 통신 포트를 자동으로 엽니다.
* **현실(장애물):** ISP(통신사)가 UPnP를 막아두었거나, 이중 NAT(아파트 공용망 등) 환경에서는 UPnP가 실패합니다.
* **해결 (홀 펀칭):** WebRTC의 기본 STUN 서버(구글 무료 서버 등)를 이용해 양방향에서 동시에 패킷을 뚫는 홀 펀칭(ICE Candidate 교환)을 시도합니다.

### Step 3. 릴레이(Relay) 서버 폴백 (최후의 수단)
* 대칭형 NAT 등 홀 펀칭이 수학적으로 불가능한 환경을 위한 우회로.
* **보수적 성공률 산정:** P2P 직접 연결 성공률을 최대 85~90%로 잡고, 나머지 10~15%의 유저는 우리가 구축한 초경량 TURN/Relay 서버를 경유하도록 설계합니다. (비용 청구 모델과 연계)

## 4. mTLS (상호 TLS 인증)
* 시그널링이 끝나고 기기 간 P2P 소켓이 열리는 순간, 서버가 클라이언트를, 클라이언트가 서버를 동시에 검증하는 mTLS를 도입합니다.
* 인증서를 최초 페어링 시 교환하여 기기에 저장하며, 이후로는 훔쳐낸 IP를 통해 악의적으로 포트를 스캐닝해도 인증서가 없으면 TCP/UDP 계층에서 즉시 연결을 드랍(Drop)시킵니다.
